# Decision — Admin UI v1: Single-File Node-Graph Concept Map

**Date:** 2026-04-19
**Status:** Accepted
**Reversibility:** Easy (delete `admin/` folder, no downstream coupling)
**Affects:** New [admin/](../admin/) directory, [CASCADE.md](../CASCADE.md) (pointer added)

---

## Decision

Ship a single-file HTML admin UI at [admin/index.html](../admin/index.html) that renders the concept as an interactive node graph: commanders, lineages, hybrids, ages, forks, modes, and player-journey stages, connected by typed edges. Editable in-browser with drag/drop, CRUD, per-node comments, and pan/zoom. State persisted in a companion [admin/concept.json](../admin/concept.json) file plus browser localStorage. User edits in-browser and exports JSON; Claude reads `concept.json` to pick up the user's changes.

## Context

The user (PM hat) wanted a visual audit surface — "family tree chart that shows how everything is connected and buttons to drag and drop or add/replace/edit/modify/comment." MD docs are strong for narrative but weak for connections and at-a-glance auditing. A visual layer lets the user see commander-to-lineage tilts, lineage-to-hybrid parents, age chains, and player-journey flow at once, and edit any of it without touching the MD cascade.

## Alternatives considered

- **Option A — MD-only (status quo).** Rejected: no visual connection map; at-a-glance audit impossible.
- **Option B — Library-based (vis-network / cytoscape / D3 via CDN).** Rejected: requires internet for CDN or a build step; offline operation failed the "double-click to open" bar.
- **Option C — Server-backed app (Node/Python + SQLite).** Rejected: over-engineered for concept phase; two-dev team has no appetite for running a server per session.
- **Option D (chosen) — Single self-contained HTML + JSON sidecar.** Zero deps, zero server, works from `file://`, export/import cycle bridges browser→disk for Claude to read.

## Reason chosen

Matches the user's "go big or go home for the UI" directive while preserving the two-dev, concept-phase discipline: no toolchain, no server, portable single files, human-readable JSON. The export/import cycle is the simplest reliable way to move state between the browser and the repo without breaking Claude's ability to read a plain file. MD remains narrative truth; JSON mirrors structure. Drift is managed by convention (naming → MD, structure → JSON) rather than by tooling.

3x debug loop synthesis:

- **Loop 1 critique:** "Export/import is clunky; users will forget to export and lose state / out-of-sync with repo."
- **Loop 2 steelman:** "localStorage auto-save prevents data loss; a clear unsaved-indicator in the status bar nudges export; the MD cascade is the narrative anchor so JSON drift is recoverable; for concept phase, clunky-but-simple beats elegant-but-fragile."
- **Loop 3 synthesis:** ship v1 with auto-save + unsaved indicator + session-notes field for routed questions. Revisit if drift becomes painful in practice.

## Reversibility note

Easy. `admin/` has no upstream consumers — removing it doesn't break the MD cascade. If replaced (e.g., by a library-backed tool later), the data is portable (plain JSON), so the upgrade path is clean.
